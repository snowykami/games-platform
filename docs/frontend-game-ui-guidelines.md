## 核心原则

- 游戏页必须像游戏，而不是管理后台。
- 通用按钮必须复用 `web/src/shared/components/ui/button.tsx` 中的固定 variant，不在业务页面临时拼接按钮背景色和文字色。
- 当前通用按钮只保留少量稳定语义：`default` 为深墨主按钮，`secondary`/`outline` 为米白高对比按钮，`ghost` 为低强调文本按钮，`destructive` 为危险操作按钮。
- 大厅可以保持清晰的信息卡片布局；进入具体游戏后，第一视觉应是游戏场景本身。
- 只要进入了具体游戏 slug 页面，视觉风格就必须服从该游戏本身。大厅的卡片、列表、浅色信息架构只适用于游戏总览，不应延续到游戏房间、准备页或对局页。
- 具体游戏页面可以参考 `/Users/sfkm/Downloads/xiangqi/` 这类成熟游戏实现：进入游戏后整体氛围、背景、控件、布局和交互都应围绕游戏主题重建，而不是复用大厅样式。
- 从大厅进入联机游戏时，第一屏必须是创建/加入房间页面，而不是直接进入对局。
- 带 `?room=` 的链接应进入房间准备页，展示玩家、AI、复制链接和开始按钮；开始后才进入游戏桌。
- 创建房间时应允许选择游戏规则类型和视觉主题；当前 Uno 先实现经典规则和经典牌面，后续可扩展 DLC 规则（例如 +6、+10、叠加惩罚）和联动主题牌面。
- 牌类、棋类、桌游类页面优先构建“桌面/棋盘/牌桌”核心舞台。
- 房间、日志、操作按钮是辅助信息，不应抢占游戏舞台。
- UI 要保留可扫读信息：当前回合、当前颜色/阵营、玩家身份、房间号、可执行动作。
- 游戏页应优先适配两类视口：电脑横屏和手机竖屏。其他尺寸允许渐进兼容，但不能牺牲这两类核心体验。
- 游戏页应尽量单屏完成主要操作。桌面横屏原则上不应出现整页滚动；手机竖屏只允许局部内容滚动，例如手牌区内部滚动。
- 游戏规则不能凭印象设计，要尽量上网搜索官方信息完成规则细节，避免设计出和真实明显不符的玩法。
- 涉及到美术的，可以使用生图模型，例如狼人杀人物卡，例如Codex可以使用GPT Image类模型生图来实现，不要搞一些看起来很奇怪的东西，风格要统一。另外注意图片资源大小限制，避免过大导致加载慢。

## Uno 页面规范

Uno 的页面流程：

- `/games/uno`：创建/加入房间。
- `/games/uno?room=ROOMID`：房间准备页。
- 点击开始后：进入圆桌牌桌。

Uno 页面采用“沉浸式圆桌牌桌”：

- 深色背景，使用多色斜向底色或暗色渐变承托 Uno 的红、黄、绿、蓝四色。
- 顶部是紧凑工具条：复制链接、状态、回合、方向、房间号、重开。
- 主体是圆桌舞台：
  - 椭圆绿色毡面居中。
  - 中央放摸牌堆、弃牌堆、当前颜色、剩余牌数和回合提示。
  - 玩家座位沿圆桌分布。
  - 当前玩家座位需要醒目的发光高亮，不能只依赖细微边框变化。
  - 对手手牌用背面小牌堆表示，不只显示数字。
  - 自己的座位可展示迷你明牌，但主要交互仍放在底部手牌区。
- 底部是玩家控制台：
  - 自己的完整手牌横向/换行排列。
  - 可出牌必须有明确视觉高亮。
  - 出牌采用两段式确认：第一次点击可出的牌时，牌从手牌区域向上浮起约自身高度的 25%；再次点击该牌才提交出牌动作。
  - 万能牌颜色选择使用四色色块。
  - 摸牌、出牌、重开等按钮需要明确禁用态。
- 日志默认折叠、弱化或移动到非主流程区域，避免喧宾夺主。
- AI 玩家必须和真人玩家共用座位、手牌数量、当前回合高亮等表现，仅用小徽标区分身份。
- 状态栏优先使用图形表达：当前颜色显示为圆形色块，方向显示为方向图标。
- 功能牌牌面优先使用图标表达，例如禁、转向、变色、+2、+4，而不是只显示文字。

## 卡牌视觉规范

- Uno 卡牌使用固定比例 `2 / 3`。
- 卡牌需要有明显白色边框、圆角、中心大字和内圈装饰。
- 颜色固定：
  - 红：`#d72f2f`
  - 黄：`#f3c33c`
  - 绿：`#1f9b55`
  - 蓝：`#2768c7`
  - 万能牌：四色斜向组合
- 牌背使用四色斜向背景和中心椭圆图案。
- 手牌里的可出牌应有白色描边；被选中待确认的牌应上浮约 25% 并加更醒目的描边。
- 禁用牌降低透明度，但仍保持可识别。

## 响应式规范

- 桌面横屏优先完整圆桌布局，并应尽量控制在 `100svh` 内：
  - 页面使用固定高度舞台，不让 body 产生主滚动条。
  - 布局建议为标题、工具条、游戏舞台、玩家控制台四段。
  - 游戏舞台使用 `minmax(0, 1fr)` 占据剩余空间。
  - 工具条必须可换行，但不能把游戏舞台挤到不可玩。
  - 日志和次要说明不常驻占用纵向空间。
- 移动端不得把桌面侧栏硬塞进首屏；应让牌桌、手牌、状态自然纵向排列。
- 手机竖屏优先单列压缩布局，并应尽量控制在 `100svh` 内：
  - 标题缩小，保留品牌识别即可。
  - 工具条使用小按钮和小状态胶囊，可换行。
  - 圆桌高度使用视口剩余空间，不能固定成桌面高度。
  - 手牌区允许内部滚动，避免整页滚动。
  - 非关键日志默认隐藏。
- 圆桌、座位、中心牌堆、手牌都必须有稳定尺寸，不能因为文字或牌数跳动导致布局明显位移。
- 移动端座位卡、牌背和中心牌堆要缩小，但不能重叠。
- 每次重做游戏 UI 后必须截图检查：
  - 电脑横屏游戏页，例如 `1440x900` 或 `1366x768`。
  - 手机竖屏游戏页，例如 `390x900`。
  - 至少一个带 `?room=` 的 URL。

## AI 玩家制度

参考项目已包含 AI 玩家入口和 AI 行动流。当前平台也应把 AI 玩家作为正式制度，而不是临时演示数据。

- AI 玩家是房间玩家的一种，和真人玩家共享座位、手牌数量、回合、胜负等状态。
- AI 玩家需要有明确身份标识，例如 `AI` 徽标。
- 添加 AI 是慢操作，必须立即禁用按钮并显示 `生成中...` 或 loading 状态，成功或失败后恢复。
- AI 的昵称、性格、说话风格由全局 LLM provider 生成，配置来自 `LLM_API`、`LLM_MODEL` 和 `LLM_TOKEN`。
- MVP 阶段 AI 可以使用规则驱动策略：
  - 有可出牌时选择第一张或按简单启发式选择。
  - 无可出牌时摸牌。
  - 万能牌选择自己手牌中最多的颜色。
- AI 决策必须走和真人相同的服务端动作校验路径。
- 联机阶段由房主添加 AI，服务端负责 AI 自动行动。
- AI 行动应进入公开 action log，方便未来做出牌/摸牌动画。
- 不在 MVP 中引入大模型逐回合决策，避免响应慢、成本高和行为不可控。

## 技术约定

- React 组件负责渲染，规则状态放在独立 engine 中。
- 游戏规则与 UI 分离，方便未来迁移到服务端权威状态。
- AI 决策也放在规则/服务层，不写死在展示组件里。
- Tailwind 可以用于页面布局；游戏专属复杂视觉可以使用少量语义 CSS 类。
- shadcn/ui 适合大厅、表单、弹窗、菜单等通用界面；具体游戏牌桌可以使用定制 DOM。
- 不引入第三方动画库，优先 CSS transition/animation。
- 不把参考项目源码直接复制进当前项目。

## 验证清单

默认验证策略：

- 纯文档、规范、文案类改动，如果主人没有指出具体问题，不需要跑测试、不需要截图查看。
- 只有改到可执行代码、样式布局、规则逻辑、路由流程或主人明确说有问题时，才运行对应验证。
- UI 相关改动优先跑 `pnpm --dir web verify:ui`；规则逻辑改动优先跑单元测试；后端改动优先跑 Go 测试。

常用验证命令：

- `pnpm --dir web lint`
- `pnpm --dir web test`
- `pnpm --dir web build`
- `pnpm --dir web verify:ui`
- `go test ./server/...`

# 新游戏功能检查表

新增或重构游戏时使用本清单。它用于保证游戏不是只做一个页面，而是作为平台完整能力交付。

## 产品契约

- Slug：稳定小写路由，例如 `uno`、`gomoku`、`xiangqi`、`mahjong`。
- 展示名：中文或英文名称，与平台 i18n 一致。
- 状态：可玩、规划中、实验性或仅本地。
- 玩家人数：最小/最大人数、AI 槽位、是否支持旁观。
- 模式：仅本地、仅联机或两者都支持。
- 房间流程：创建、输入房间码加入、`?room=` 加入、复制邀请链接、添加 AI、开始/重开。
- 身份：未鉴权分享链接进入登录页；游客登录保留对局数据，但其他玩家只看到游客展示名。
- 持久化：明确当前是否需要房间快照、动作记录和比赛结果存储，或只保留后续 hook。

## 规则契约

- 状态模型：公开状态、每玩家私有状态、派生 UI 状态。
- 回合模型：当前玩家、方向/顺序、同时阶段、响应/声明、待处理惩罚。
- 合法动作：枚举前后端和 AI 都使用的 action id。
- 非法动作：服务端拒绝并返回清晰错误。
- 结束条件：胜利、平局、认输、超时、无合法动作、断线处理。
- 随机性：牌堆、牌墙、棋子等在线游戏随机内容必须由服务端生成或洗牌。
- 重连：刷新后能从服务端房间状态恢复视图。

## 扩展契约

扩展点要窄而明确：

- `ruleset` / `variant`：经典、官方、房规、混乱等玩法类型。
- `modules` / `dlc`：特殊牌、额外计分、自定义棋盘等可选模块。
- `theme`：纯视觉卡面、棋子或牌面皮肤。主题不能改变规则，除非明确配套 ruleset。
- `aiLevel`：`beginner`、`normal`、`master`、`ai`。
- `aiPersonality`：风味和策略偏好，不拥有规则权威。

不要提前完整实现未经要求的变体；低成本保留数据结构和 UI 插槽即可。

## 后端形态

Go 在线游戏包优先拆分：

- `types.go`：领域类型、房间/玩家/action/public state DTO。
- `manager.go`：房间生命周期和协调逻辑。
- `rules_*.go`：规则转换、合法动作、结算和胜负判定。
- `view_*.go`：公开/私有视图和 AI context 投影。
- `ai_*.go`：AI 状态、action alias、fallback、prompt/context。
- `http.go`：REST 路由和请求解码。
- `hub.go`：WebSocket 订阅、广播和连接生命周期。

服务端拥有权威状态。HTTP/WebSocket handler 解码请求、调用规则入口、广播结果和调度 AI。

## 前端形态

每个游戏一个目录：

- `types.ts`：前端领域类型。
- `online.ts`：API schema/client。
- `use<Game>Room.ts`：WebSocket + REST 状态 hook。
- `<Game>RoomGate.tsx`：创建/加入/准备页。
- `<Game>Page.tsx`：活跃游戏界面。
- `engine.ts`：本地模式或纯规则工具。

房间页和对局页都应具备游戏自己的视觉风格，而不是复用大厅卡片风格。

## 完成标准

- 游戏出现在大厅，具备图像/图标元信息。
- `/games/:slug` 可打开。
- 联机游戏支持 `?room=`。
- AI 添加流程支持配置的 level。
- 当前回合和合法动作在 UI 中明确。
- 桌面横屏和手机竖屏可用。
- 后端拒绝非法动作。
- LLM 返回动作必须校验 legal action id。
- 用户可见文案 i18n-ready。
